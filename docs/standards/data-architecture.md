# Data Architecture Standard — Grails / GORM

## 1. Principles

- **The database is the system of record** for this application's own
  data. If this app references data owned by another system, store the
  reference (an ID/key), not a copy — resolve details through that
  system's API/integration point, not a stale local mirror.
- **One schema, module/domain-owned tables.** Each domain area owns its
  tables; avoid cross-domain foreign keys where the app has clear module
  boundaries — reference by ID and resolve through the service layer, not
  a database-level join across boundaries.
- **Migrations only.** All schema changes go through versioned migrations
  (Flyway, or the Grails Database Migration plugin). No relying on GORM's
  `dbCreate: update` in any shared environment (staging, production).
- **Data classification drives handling** (see §2) — decide the
  classification when the field is designed, not after an incident.

## 2. Data classification

| Class | Examples | Handling |
|-------|----------|----------|
| **Sensitive personal data** | national/government ID, exact financial details, health/legal status | Encrypted at rest, strict access control, full audit, never logged |
| **Personal data** | name, phone, address, email | Encrypted at rest where feasible, minimized, masked in logs/UI where not needed |
| **Sensitive operational** | payment references, issued certificates/licenses | Integrity-protected, audited |
| **Internal** | configuration, catalogs, internal reference data | Standard access controls |
| **Public** | published/non-sensitive reference data | Cacheable, no special handling |

High-sensitivity identifiers used only for matching (not display) may be
stored as a keyed hash instead of the raw value.

## 3. Naming & modeling conventions

- Tables: `snake_case`, plural (GORM's default pluralized mapping is fine;
  override explicitly with `mapping { table '...' }` if a different name
  is needed).
- Primary keys: prefer a UUID (v7 for index locality) generated by the
  application over a bare auto-increment where the ID may ever be exposed
  externally (URLs, APIs) — a sequential integer ID leaks record count and
  invites enumeration.
- Timestamps: `createdAt` / `updatedAt` (`timestamptz`, stored UTC).
  Business-meaningful dates (a due date, an expiry date) are modeled
  separately as plain dates, not conflated with audit timestamps.
- Money: store as an integer in minor units (cents, fils, etc.) plus an
  ISO-4217 currency code — never a floating-point type.
- Every mutable domain class has a `version` field (GORM's built-in
  optimistic locking) — see `docs/standards/code-style.md`'s concurrency
  guidance.
- Soft-delete only where an audit/legal requirement demands the row
  persist; otherwise hard-delete with an audit trail (§4) rather than
  carrying a `deleted` flag everywhere queries have to remember to filter.

## 4. Auditing

- State-changing operations on sensitive/regulated data write an audit
  record: who, what, when, and enough of a before/after summary to
  reconstruct the change — append-only, never mutated or deleted.
- Prefer a dedicated audit table/mechanism over relying on `updatedAt`
  alone, which only shows the latest state, not the history of changes.

## 5. Migrations workflow

```text
grails-app/migrations/           (Grails Database Migration plugin)
  001_baseline.groovy
  002_add_condition_field.groovy
  ...
# or, if using Flyway directly:
src/main/resources/db/migration/
  V1__baseline.sql
  V2__add_condition_field.sql
```

- One logical change per migration. Never edit a migration that has run
  in a shared environment — add a new one, even to fix a mistake in the
  previous one.
- Migrations are reviewed like code. A migration that drops or renames a
  column, or backfills at scale, gets an ADR (`docs/adr/`) describing the
  rollback plan.
- Seed/reference data ships via its own migration, not inserted manually
  post-deploy.

## 6. Reporting and read-heavy queries

For dashboards, KPIs, or any read pattern that would otherwise put load on
the transactional tables, read from a dedicated read model (a
materialized view, a reporting-specific query, or a separate schema) —
not directly against write-path tables. Aggregate/anonymize sensitive data
before it reaches a report; a report should never be the accidental
easiest path to bulk sensitive data.
